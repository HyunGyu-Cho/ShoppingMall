# 설계원칙

## 설계방향성 평가 및 개선방안

### 1. 총평

현재 설계 방향은 좋다. 단순 상품 CRUD 쇼핑몰이 아니라 **주문, 결제, 재고 정합성**을 중심으로 시스템을 정의하고 있다. 특히 다음 관점이 명확하다.

- 주문과 상품을 직접 다대다로 묶지 않고 `order_item`에 주문 당시 상품 스냅샷을 저장한다.
- 상품 정보와 재고 정보를 분리해 재고 동시성 제어를 별도 관심사로 둔다.
- 재고 차감/예약을 조건부 `UPDATE`와 row count로 판단한다.
- 결제 승인, PG 콜백, 결제 취소에 멱등성을 고려한다.
- 주문, 결제, 배송을 상태 머신으로 다룬다.
- 조회 성능을 위해 cursor pagination, DTO 조회, 복합 인덱스를 고려한다.
- 장애 복구를 위해 outbox, event log, reconciliation batch, batch failure를 설계에 포함했다.

이 방향은 포트폴리오용 쇼핑몰 프로젝트로도 설득력이 있다. 흔한 CRUD 예제와 다르게 운영 환경에서 실제로 깨지는 지점을 먼저 잡고 있기 때문이다.

다만 현재 문서는 운영 전체 아키텍처와 1차 MVP가 섞여 있다. 설계 자체는 넓고 깊지만, 구현 단계에서는 범위를 더 강하게 잘라야 한다. 지금 가장 중요한 원칙은 **운영 가능한 최종 그림은 유지하되, 1차 구현은 주문·결제·재고 정합성 검증에 집중하는 것**이다.

---

### 2. 잘 잡힌 설계 원칙

#### 2-1. 도메인 중심 설계 방향

현재 설계는 `Member`, `Product`, `Inventory`, `Order`, `OrderItem`, `Payment`, `Delivery`, `Review`를 핵심 도메인으로 분리한다. 이 분리는 타당하다.

특히 `Product`와 `Inventory`를 분리한 점이 좋다. 상품 정보는 조회가 많고 변경 빈도가 낮지만, 재고는 주문마다 변경되고 동시성 제어가 필요하다. 두 관심사를 한 테이블에 섞으면 구현은 쉬워지지만, 재고 예약, 결제 실패 복구, 선착순 주문 같은 요구사항에서 금방 복잡해진다.

#### 2-2. 주문 스냅샷 설계

`order_item`에 주문 당시 상품명, 가격, 옵션, 할인 금액을 저장하는 방향은 맞다.

상품명이나 가격이 나중에 바뀌어도 과거 주문 내역은 바뀌면 안 된다. 따라서 주문 내역은 상품 테이블을 실시간 참조해서 보여주는 구조가 아니라, 주문 시점의 데이터를 보존하는 구조여야 한다.

#### 2-3. 동시성 제어 방향

재고를 애플리케이션에서 먼저 조회한 뒤 차감하는 방식이 아니라, DB 조건부 `UPDATE`로 처리하려는 방향은 실용적이다.

핵심은 다음 형태다.

```sql
UPDATE inventory
SET reserved_quantity = reserved_quantity + :quantity,
    available_quantity = available_quantity - :quantity
WHERE product_id = :productId
  AND available_quantity >= :quantity;
```

성공 여부는 변경 row 수로 판단한다. 이 방식은 비관적 락을 직접 오래 잡는 방식보다 단순하고, 일반적인 쇼핑몰 재고 차감에는 충분히 강하다.

#### 2-4. 멱등성 고려

결제 승인 요청, PG 콜백, 결제 취소 요청은 중복될 수 있다. 이 점을 설계 초반부터 반영한 것은 좋다.

`payment_key`, `idempotency_key`, `pg_transaction_id`에 UNIQUE 제약을 두고, 이미 처리된 요청이면 기존 결과를 반환하는 방향은 실무적인 설계다.

#### 2-5. 장애 복구와 운영 관점

`outbox_event`, `order_event_log`, `batch_job_execution`, `batch_failure`를 고려한 점은 좋다. 단순 구현을 넘어 장애 발생 후 어떤 주문을 재처리해야 하는지 추적하려는 의도가 있다.

이 프로젝트의 강점은 여기다. 기능을 많이 만드는 것보다, 주문·결제·재고가 깨졌을 때 복구 가능한 구조를 고민했다는 점이 더 중요하다.

### 3. 주요 개선 필요사항

#### 3-1. 최종 아키텍처와 MVP 범위를 더 분리해야 한다

문서에는 Redis, MessageQueue, ObjectStorage, BatchServer, Outbox, Monitoring까지 포함되어 있다. 최종 아키텍처로는 좋지만, 1차 구현에서 모두 구현하려고 하면 범위가 과도하다.

개선 원칙:

- 1차는 **모듈러 모놀리스 + 단일 RDBMS**로 간다.
- 패키지는 도메인별로 나누되, 배포 단위는 하나로 유지한다.
- Redis, MQ, 실제 PG, 실제 배송사 연동은 2차 이후로 미룬다.
- Outbox는 1차에서 제외한다면, 대신 `order_event_log`로 상태 변경 이력을 남긴다.

1차 MVP의 성공 기준은 기능 개수가 아니라 다음 세 가지다.

- 동시에 주문해도 재고가 음수가 되지 않는다.
- 결제 성공/실패에 따라 주문 상태와 재고 상태가 일관된다.
- 중복 결제 승인/콜백이 들어와도 한 번만 처리된다.

#### 3-2. 결제 승인 트랜잭션 경계를 더 명확히 해야 한다

문서에서는 외부 PG 호출을 DB 트랜잭션 밖에서 처리해야 한다는 원칙이 있다. 이 원칙은 맞다. 다만 결제 승인 시퀀스에서는 PG 승인 호출과 DB 상태 변경이 하나의 흐름으로 붙어 있어, 실제 구현 시 긴 트랜잭션으로 이어질 위험이 있다.

개선 원칙:

1. DB에서 주문과 결제 준비 상태를 확인한다.
2. DB 트랜잭션을 오래 열어두지 않는다.
3. PG 승인 API를 호출한다.
4. 짧은 DB 트랜잭션에서 `payment APPROVED`, `order PAID`, 예약 재고 확정을 처리한다.
5. 중복 요청이면 기존 승인 결과를 반환한다.

즉, 외부 API 호출 중에는 DB 커넥션과 row lock을 오래 점유하지 않아야 한다.

#### 3-3. 재고 수량의 기준 값을 하나로 정해야 한다

현재 `inventory`에는 `total_quantity`, `reserved_quantity`, `available_quantity`가 있다. 이 구조는 이해하기 쉽지만, 세 값이 서로 불일치할 위험이 있다.

개선 원칙:

- 기본 불변식은 `total_quantity = reserved_quantity + available_quantity`로 둔다.
- `available_quantity`를 저장할지, 계산값으로 둘지 결정한다.
- 저장한다면 모든 재고 변경 SQL에서 위 불변식이 깨지지 않도록 DB `CHECK`와 테스트를 둔다.
- 결제 성공 시 `reserved_quantity`와 `total_quantity`를 함께 줄이는 정책이 맞는지 명확히 문서화한다.

1차 구현에서는 `available_quantity`를 저장해도 된다. 대신 모든 변경은 전용 Repository 메서드로만 수행하고, 서비스에서 임의로 수량을 수정하지 않는 원칙이 필요하다.

#### 3-4. 주문 상태 전이표를 더 엄격하게 정리해야 한다

현재 주문 상태 목록은 풍부하지만, 일부 전이 표현이 선형 흐름처럼 보여 오해될 수 있다. 예를 들어 `PENDING_PAYMENT → PAID → PAYMENT_FAILED → CANCELLED`처럼 읽히면, 결제 완료 후 결제 실패로 이동 가능한 것처럼 보일 수 있다.

개선 원칙:

- 상태 전이는 표로 관리한다.
- 각 API는 허용된 현재 상태에서만 실행된다.
- 상태 변경은 Controller가 아니라 도메인 메서드 또는 전용 상태 전이 서비스에서 검증한다.

예시:

| 현재 상태 | 허용 전이 | 트리거 |
| --- | --- | --- |
| `PENDING_PAYMENT` | `PAID` | 결제 승인 |
| `PENDING_PAYMENT` | `PAYMENT_FAILED` | 결제 실패 |
| `PENDING_PAYMENT` | `CANCELLED` | 결제 대기 만료 |
| `PAID` | `PREPARING` | 배송 준비 |
| `PAID` | `CANCELLED` | 배송 전 주문 취소 |
| `PREPARING` | `SHIPPED` | 배송 시작 |
| `SHIPPED` | `DELIVERED` | 배송 완료 |
| `DELIVERED` | `CONFIRMED` | 구매 확정 |

#### 3-5. 멱등성 저장 모델을 구체화해야 한다

`idempotency_key`를 UNIQUE로 두는 방향은 좋지만, 실제 운영에서는 키만 저장하면 부족하다.

개선 원칙:

- `idempotency_key`와 함께 요청 payload hash를 저장한다.
- 같은 key로 다른 payload가 들어오면 실패 처리한다.
- 처리 상태를 `PROCESSING`, `SUCCEEDED`, `FAILED`로 구분한다.
- 성공한 요청은 기존 응답을 재사용할 수 있게 결과 요약을 저장한다.
- 결제 취소도 별도 `cancel_idempotency_key` 또는 `payment_cancel` 테이블로 관리한다.

1차 MVP에서는 간단히 `payment.idempotency_key UNIQUE`로 시작해도 된다. 다만 README와 설계 문서에는 위 확장 방향을 남기는 것이 좋다.

#### 3-6. Outbox 제외 여부를 명확히 해야 한다

구현 계획에서는 1차 MVP에서 Outbox를 제외한다고 되어 있는데, API/시퀀스 문서에서는 결제 완료 시 `outbox_event` 저장이 등장한다.

둘 중 하나로 정리해야 한다.

- 1차 MVP에서 Outbox를 제외한다면: 결제 완료 후 알림/통계 이벤트는 구현하지 않고, `order_event_log`만 남긴다.
- 1차 MVP에서 Outbox를 포함한다면: MQ 없이 DB outbox 테이블과 polling publisher까지만 구현한다.

추천은 1차에서는 Outbox를 제외하는 것이다. 대신 주문/결제 상태 변경 로그를 충실히 남기고, 2차에서 Outbox를 추가하는 편이 낫다.

#### 3-7. DB 제약조건의 세부 위험을 보완해야 한다

ERD는 전반적으로 좋지만 다음 부분은 구현 전 점검이 필요하다.

- 금액 컬럼은 `INT`보다 `BIGINT` 또는 `DECIMAL`이 안전하다.
- `pg_transaction_id UNIQUE`가 nullable일 때 DB별 UNIQUE 동작을 확인해야 한다.
- soft delete가 있는 테이블의 UNIQUE 제약은 탈퇴/삭제 후 재가입 정책과 충돌할 수 있다.
- `JSON` 타입은 DBMS별 호환성이 다르므로 MySQL/PostgreSQL 기준을 명확히 해야 한다.
- FK를 강하게 둘 테이블과 운영상 느슨하게 둘 로그성 테이블을 구분해야 한다.

#### 3-8. 테스트 기준을 더 측정 가능하게 만들면 좋다

현재 테스트 방향은 좋다. 여기에 통과 기준을 더 명확히 두면 구현 품질이 올라간다.

추가할 테스트:

- 재고 1개 상품에 100명 동시 주문 시 성공 1건, 실패 99건, 최종 재고 0 검증
- 같은 `Idempotency-Key`로 결제 승인 10번 호출 시 승인 1건만 생성되는지 검증
- PG 콜백 중복 수신 시 주문/결제/재고가 중복 변경되지 않는지 검증
- 결제 실패 시 예약 재고가 정확히 해제되는지 검증
- 주문 취소와 결제 콜백이 동시에 들어오는 경우 상태가 깨지지 않는지 검증
- 주문 목록 조회에서 N+1이 발생하지 않는지 쿼리 수 검증
- cursor pagination에서 중복/누락이 없는지 검증

### 4. 권장 구현 원칙

#### 4-1. 1차 구현 원칙

1차 구현은 다음 원칙으로 간다.

- 배포 단위는 하나의 Spring Boot 애플리케이션으로 유지한다.
- 도메인 패키지는 분리한다.
- 실제 PG 대신 Mock 결제 API를 사용한다.
- 실제 배송사 연동은 제외한다.
- Redis, MQ, ObjectStorage는 제외한다.
- 핵심은 주문 생성, 재고 예약, 결제 성공/실패, 주문 취소, 리뷰 중복 방지다.

#### 4-2. 패키지 기준

권장 패키지 구조:

```text
com.example.myshop
├── member
├── product
├── inventory
├── order
├── payment
├── delivery
├── review
├── common
└── support
```

초기부터 계층을 과도하게 나누기보다, 도메인별 응집도를 우선한다.

#### 4-3. 트랜잭션 원칙

- DB 정합성이 필요한 변경만 트랜잭션 안에서 처리한다.
- 외부 API 호출, 이메일, SMS, 파일 업로드는 트랜잭션 밖에서 처리한다.
- 재고 변경은 전용 Repository 메서드를 통해서만 수행한다.
- 주문 상태 변경은 도메인 메서드로 검증한다.
- 결제 승인/콜백/취소는 멱등하게 만든다.

#### 4-4. API 원칙

- 사용자 API와 관리자 API를 분리한다.
- 응답 포맷은 일관되게 유지한다.
- 오류 응답에는 `error.code`, `error.message`, `traceId`를 포함한다.
- 목록 조회는 기본적으로 cursor pagination을 사용한다.
- 상태 변경 API는 멱등성 또는 중복 요청 방어 정책을 가진다.

### 5. 구현 우선순위

#### 1순위

- `member`, `product`, `inventory` 테이블 및 Entity
- 상품 등록/목록/상세 조회
- 재고 예약 조건부 `UPDATE`
- 주문 생성
- `order_item` 주문 스냅샷 저장
- `delivery` 기본 정보 저장
- `payment READY` 생성

#### 2순위

- Mock 결제 승인
- `payment APPROVED`
- `order PAID`
- 예약 재고 확정
- Mock 결제 실패
- `payment FAILED`
- `order PAYMENT_FAILED`
- 예약 재고 해제

#### 3순위

- 주문 취소
- 배송 시작 전 취소 가능 조건
- 결제 취소 Mock 처리
- 재고 복구
- 주문 이벤트 로그 저장

#### 4순위

- 리뷰 작성
- 구매자 검증
- `order_item_id` UNIQUE로 중복 리뷰 방지
- 리뷰 목록 cursor pagination

#### 5순위

- 주문 목록 조회 최적화
- 상품 목록 조회 최적화
- N+1 제거
- 인덱스 적용 전/후 비교
- 동시성 테스트 결과 README 정리

### 6. 최종 설계 원칙 요약

이 프로젝트의 핵심 설계 원칙은 다음이다.

1. 상품 CRUD보다 주문·결제·재고 정합성을 우선한다.
2. 재고 변경은 조건부 `UPDATE`로 원자적으로 처리한다.
3. 결제와 콜백은 반드시 멱등하게 처리한다.
4. 외부 API 호출을 DB 트랜잭션 안에 넣지 않는다.
5. 주문 상태, 결제 상태, 배송 상태는 명시적인 상태 전이 규칙으로 관리한다.
6. 주문 내역은 상품 현재값이 아니라 주문 당시 스냅샷을 저장한다.
7. 중복 리뷰, 중복 결제 같은 race condition은 애플리케이션 검증뿐 아니라 DB 제약조건으로도 막는다.
8. 목록 조회는 offset보다 cursor pagination을 우선한다.
9. 장애 복구를 위해 상태 변경 이력과 실패 이력을 남긴다.
10. 1차 MVP는 넓게 만들지 말고, 핵심 정합성 흐름을 테스트로 증명한다.

현재 설계는 방향이 좋다. 개선의 핵심은 더 많은 기능을 추가하는 것이 아니라, **MVP 범위를 좁히고 상태 전이, 트랜잭션 경계, 멱등성 저장 방식, 재고 불변식을 더 엄격하게 만드는 것**이다.
