# 06-29 개발 진행 정리

## 1. 개발 방향 정리

문서 검토 결과, 1차 MVP는 단순 상품 CRUD가 아니라 주문, 결제, 재고 정합성을 검증하는 최소 주문 시스템으로 진행하기로 정리했다.

권장 개발 순서는 다음으로 확정했다.

1. 현재 빌드 복구
2. ERD 기준 Entity / Enum 기본 골격 작성
3. API 명세 기준 Request / Response DTO 작성
4. 시퀀스 다이어그램 기준 Service 유스케이스 작성
5. Service에서 필요한 Repository 메서드 작성
6. Controller에서 API와 Service 연결
7. 테스트 작성
8. 조회 성능 / 인덱스 / 동시성 보완

이번 작업에서는 1번부터 3번까지 진행했다.

## 2. 빌드 복구

`ProductService`에서 `ProductSort` import 경로가 잘못되어 컴파일이 실패하고 있었다.

수정 전:

```java
import shopingmall.myshop.product.dto.ProductSort;
```

수정 후:

```java
import shopingmall.myshop.product.dto.enums.ProductSort;
```

수정 후 `compileJava`를 실행해 메인 소스 컴파일 성공을 확인했다.

```text
./gradlew.bat compileJava
BUILD SUCCESSFUL
```

## 3. ERD 기준 Entity / Enum 기본 골격 작성

기존에 파일은 있었지만 내용이 비어 있던 핵심 도메인 파일들을 ERD 기준으로 채웠다.

### Inventory

작성 파일:

- `myshop/src/main/java/shopingmall/myshop/inventory/domain/Inventory.java`

주요 필드:

- `productId`
- `product`
- `totalQuantity`
- `reservedQuantity`
- `availableQuantity`
- `version`
- `updatedAt`

현재 불변식:

```text
totalQuantity = reservedQuantity + availableQuantity
```

### Order

작성 파일:

- `myshop/src/main/java/shopingmall/myshop/order/domain/Order.java`
- `myshop/src/main/java/shopingmall/myshop/order/domain/OrderItem.java`
- `myshop/src/main/java/shopingmall/myshop/order/domain/OrderStatus.java`
- `myshop/src/main/java/shopingmall/myshop/order/domain/OrderEventLog.java`

`OrderStatus`:

```java
PENDING_PAYMENT,
PAID,
PAYMENT_FAILED,
CANCELLED,
DELIVERED,
CONFIRMED
```

`Order`에는 주문번호, 회원, 주문 상태, 금액, 주문 시각, 주문 상품 목록을 추가했다.

`OrderItem`에는 주문 당시 상품 스냅샷을 저장하도록 했다.

주요 스냅샷 필드:

- `productNameSnapshot`
- `orderPrice`
- `quantity`
- `discountAmount`
- `optionNameSnapshot`

### Delivery

작성 파일:

- `myshop/src/main/java/shopingmall/myshop/delivery/domain/Delivery.java`
- `myshop/src/main/java/shopingmall/myshop/delivery/domain/DeliveryStatus.java`

`DeliveryStatus`:

```java
READY,
SHIPPED,
DELIVERED
```

1차 MVP에서는 기본 배송 상태를 `READY`로 사용한다.

### Payment

작성 파일:

- `myshop/src/main/java/shopingmall/myshop/payment/domain/Payment.java`
- `myshop/src/main/java/shopingmall/myshop/payment/domain/PaymentStatus.java`

`PaymentStatus`:

```java
READY,
APPROVED,
FAILED,
CANCELLED
```

주요 필드:

- `paymentKey`
- `idempotencyKey`
- `paymentStatus`
- `requestedAmount`
- `approvedAmount`
- `cancelledAmount`
- `approvedAt`
- `failedAt`
- `cancelledAt`
- `failureCode`
- `failureMessage`

### Review

작성 파일:

- `myshop/src/main/java/shopingmall/myshop/review/domain/Review.java`

주요 제약:

- `order_item_id` UNIQUE
- `rating`은 1 이상 5 이하

리뷰는 회원, 상품, 주문 상품을 참조하도록 기본 골격을 작성했다.

## 4. API 명세 기준 DTO 작성

API 명세를 기준으로 비어 있던 Request / Response DTO를 채웠다.

### Product DTO

작성 / 수정 파일:

- `ProductCreateRequest.java`
- `ProductCreateReqeust.java`
- `ProductResponse.java`

`ProductCreateReqeust`는 오타가 있는 기존 파일이라 삭제하지 않고 `@Deprecated` 처리했다.

새 코드는 `ProductCreateRequest`를 사용한다.

### Inventory DTO

작성 파일:

- `InventoryResponse.java`

응답 필드:

- `productId`
- `totalQuantity`
- `reservedQuantity`
- `availableQuantity`

### Delivery DTO

작성 파일:

- `DeliveryRequest.java`
- `DeliveryResponse.java`

### Order DTO

작성 파일:

- `CreateOrderRequest.java`
- `CreateOrderResponse.java`
- `CancelOrderRequest.java`
- `OrderListResponse.java`
- `OrderDetailResponse.java`

`CreateOrderRequest`는 주문 상품 목록과 배송 정보를 포함한다.

```java
List<Item> items
DeliveryRequest delivery
```

`CreateOrderResponse`는 주문 생성 후 결제 준비 정보를 포함한다.

```java
orderId
orderNo
orderStatus
totalPrice
discountAmount
finalPaymentAmount
paymentReady
```

### Payment DTO

작성 파일:

- `PaymentApproveRequest.java`
- `PaymentFailRequest.java`
- `PaymentCancelRequest.java`
- `PaymentResponse.java`

### Review DTO

작성 파일:

- `ReviewCreateRequest.java`
- `ReviewResponse.java`
- `ReviewListResponse.java`

리뷰 생성 요청 필드:

```java
orderItemId
rating
content
```

## 5. 현재 검증 상태

메인 소스 컴파일은 성공했다.

```text
./gradlew.bat compileJava
BUILD SUCCESSFUL
```

아직 전체 테스트는 실행하지 않았다. 이번 범위가 Entity / DTO 골격 작성까지였고, 기존 테스트는 Service 구현 상태에 따라 추가 수정이 필요할 수 있다.

## 6. 아직 하지 않은 일

다음 항목은 아직 진행하지 않았다.

- Service 유스케이스 구현
- Repository 메서드 구현
- Controller와 Service 연결
- 통합 테스트 작성
- 동시성 테스트 작성
- 조회 성능 / 인덱스 보완

## 7. 다음 작업

다음 단계는 시퀀스 다이어그램 기준으로 Service 유스케이스를 작성하는 것이다.

우선순위는 다음이 적절하다.

1. `OrderService.createOrder`
2. `InventoryService.reserve`
3. `PaymentService.approve`
4. `PaymentService.fail`
5. `OrderService.cancel`
6. `ReviewService.createReview`

이후 각 Service에서 필요한 Repository 메서드를 추가하고, Controller에서 API와 연결한다.
